## 1. 先用一句话认识项目

“星舰猎盘”是一间自动整理宏观经济数据的数字研究室。

它目前只开放了第一间研究室：`cpi-research`。这间研究室会读取美国 CPI（消费者价格指数）数据，计算价格相对上月、去年同期发生了什么变化，再把数字整理成网页上的指标卡、分项条形图、观察信号和资产影响说明。

可以把整套系统想成一家小型报社：

1. 美国劳工统计局 BLS 是“官方消息源”。
2. Python 更新器是“采编员”，负责取数据、检查数据和写稿。
3. `latest.json` 是“最新稿件”，`history.json` 是“历月合订本”。
4. React 页面是“排版员”，把稿件排成网页。
5. CSS 是“美术编辑”，决定颜色、大小和布局。
6. Vinext、Vite 和 Cloudflare Worker 是“印刷机与发行系统”。
7. 测试代码是“校对员”，防止公式和页面被改坏。
8. GitHub Actions 是“闹钟加机器人”，定时叫醒 Python 更新器。

### 1.1 它解决什么业务问题

每次 CPI 发布后，研究者通常要做四件事：

- 找到可信的官方数字；
- 计算环比、同比等指标；
- 判断能源、住房、食品等分项谁在推动变化；
- 把事实转成交易员容易阅读的摘要。

本项目把这四步做成固定流程，减少手工复制和口算造成的错误。

### 1.2 当前网页能看到什么

- 最新报告月份；
- 总 CPI、核心 CPI、住房和能源等关键指标；
- 各分项的环比条形图；
- “降温”或“观察”等规则信号；
- 交易员重点关注项；
- 美债、美元、权益和黄金的影响路径说明；
- 星舰猎盘未来模块路线图。

### 1.3 当前没有实现什么

理解“没有实现的部分”同样重要：

- 页面已有历史月份选择，未来还可以增加月份之间的直接对比；
- 登录用户可以收藏报告，但登录身份只由部署平台提供，本地匿名环境不能模拟真实账号；
- D1 已保存历史报告与收藏，当前还没有独立的数据库管理后台；
- 当前信号由固定规则生成，不是大模型自动撰写；
- 页面不是实时行情终端，不提供买卖指令；
- GitHub 定时任务更新的是仓库里的 JSON，部署平台是否自动重新发布，还取决于部署设置；
- `examples/d1/` 是数据库示例，不属于当前 CPI 正式运行链路。

### 2.3 Python 需要额外安装包吗

不需要。当前 Python 更新器只使用 Python 自带工具，例如 `json`、`urllib`、`datetime` 和 `unittest`，没有 `requirements.txt`。

### 2.4 为什么 Node 20 不能运行

Vinext 使用了 Node 22 提供的文件查找能力。Node 20 没有该能力，所以会出现：

```text
node:fs/promises does not provide an export named 'glob'
```

这不是 CPI 代码坏了，而是“发动机版本太旧”。

### 2.5 `package.json` 逐项说明

`package.json` 是整个 Node 项目的身份证、命令菜单和采购清单。

① 要实现什么：告诉 npm 项目叫什么、使用什么 Node 版本、有哪些快捷命令、需要下载哪些依赖。

② 每行/每组作用：

- `name: "xingjian-liepan"` 是程序内部名称；
- `version: "0.1.0"` 是当前软件版本；
- `private: true` 防止误把项目发布成公开 npm 软件包；
- `engines.node: ">=22.13.0"` 声明最低 Node 版本；
- `scripts` 是命令菜单：
  - `dev` 启动本地开发服务器，并把 Wrangler 日志写到项目内；
  - `build` 生成生产文件；
  - `start` 启动生产构建；
  - `test` 按顺序运行 Python 测试、TypeScript 测试、构建和 HTML 测试；使用 `&&` 表示前一步成功才执行下一步；
  - `test:python` 在 `tests/python` 查找 `test_*.py`；
  - `test:unit` 让 Node 直接测试 TypeScript 计算文件；
  - `lint` 做代码规范检查，并跳过构建产物；
  - `data:update` 启动 Python 数据更新脚本；
  - `db:generate` 根据数据库表定义生成迁移；
- `dependencies` 是程序运行时可能需要的正式依赖；
- `devDependencies` 是开发、检查、构建和部署工具；
- 每个依赖右侧的数字是锁定版本，随意升级可能产生兼容问题；
- `type: "module"` 表示 JavaScript 使用现代 `import/export` 模块写法。

③ 运行后：npm 根据这份文件理解 `npm run dev` 等命令，但真正的精确依赖树还要结合 `package-lock.json`。

## 4. 数据从哪里来，最后去了哪里

### 4.1 自动更新数据流

```text
GitHub 定时器
    ↓ 工作日北京时间 23:30 唤醒
scripts/update_cpi_data.py
    ↓ 调用
python/cpi_research/updater.py
    ↓ 通过互联网请求
BLS Public Data API
    ↓ 返回多个 CPI 月度序列
检查月份、数字和完整性
    ↓
计算环比、同比和规则信号
    ↓
lib/cpi/data/latest.json + history.json
    ↓ 数据变化时
GitHub 机器人自动 git commit + git push
```

### 4.2 页面显示数据流

```text
latest.json / history.json / D1
    ↓ latest-data.ts 加上 TypeScript 格式
page.tsx 读取 cpiReport
    ↓ React 把数组重复展开成卡片和列表
globals.css 上颜色和排版
    ↓ Vinext + Vite 构建
worker/index.ts 接收浏览器请求
    ↓
浏览器显示 http://localhost:3000
```

### 4.3 重要的数据原则

- 环比使用季节调整后的序列，减少春节、天气等固定季节因素干扰；
- 同比使用未季调序列，对比今年同月和去年同月；
- 至少需要连续 13 个月，才能计算同比；
- 所有需要同时展示的序列必须有共同的最新月份；
- 更新时间变化但数据没变时，不重写 JSON，避免每天产生无意义 Git 提交；
- 页面只展示快照，不在浏览器里临时重新计算官方数据。

## 5. 读代码前必须认识的 10 个概念

### 5.1 变量：贴了名字的盒子

```ts
const maxBar = 9.7;
```

意思是准备一个叫 `maxBar` 的盒子，里面放数字 `9.7`。`const` 表示之后不把整个盒子换成别的值。

### 5.2 函数：一台有入口和出口的小机器

```ts
function double(value: number) {
  return value * 2;
}
```

把 `3` 放进 `double`，机器返回 `6`。项目里的 `formatPercent`、`percentChange`、`build_snapshot` 都是这种机器。

### 5.3 判断：道路岔口

```ts
if (value > 0) {
  return "上涨";
}
return "没有上涨";
```

条件成立走第一条路，不成立走第二条路。

### 5.4 循环：对一筐东西逐个处理

```python
for item in observations:
    print(item)
```

就像从购物篮逐件拿商品扫码，直到全部处理完。

### 5.5 数组或列表：有顺序的清单

```json
["汽油", "能源", "住房"]
```

React 的 `.map(...)` 会把清单中的每一项变成一个页面卡片。

### 5.6 对象或字典：一张登记表

```json
{"name": "能源", "mom": -5.7}
```

`name` 和 `mom` 是栏目名，右侧是栏目内容。

### 5.7 类型：盒子上的内容标签

```ts
value: number
```

意思是 `value` 盒子应该放数字。TypeScript 会在运行前提醒“这里误放了文字”。类型不会自己产生业务数据。

### 5.8 API：餐厅点餐窗口

程序把“我要这些系列、这些年份”交给 BLS API；BLS 后台处理后，把 JSON 数据递回来。调用者不需要知道 BLS 内部数据库如何工作。

### 5.9 数据库：可查询的电子档案柜

JSON 更像一份随代码发布的文件；数据库更像有很多抽屉、可以按条件快速查找的档案柜。当前页面优先把 JSON 历史同步到 D1 并从数据库读取；D1 不可用时退回 JSON。

### 5.10 异步：下单后等待结果

网络请求和数据库查询需要时间。`async` 表示这台机器可能要等待，`await` 表示“等结果回来再继续”。

## 6. Python 数据更新代码逐份解析

Python 是当前数据链路的源头，建议先读它，再读网页。

### 6.1 `python/cpi_research/__init__.py`

① 要实现什么：把文件夹声明成一个可导入的 Python 功能包，并公开三个主要功能。

② 每行作用：

- 第 1 行是说明文字；
- 第 3 行从 `updater.py` 拿出“生成快照、抓数据、更新文件”三个函数；
- 第 5 行列出本功能包对外公开的名字。

③ 运行后：其他文件可以写 `from cpi_research import update_snapshot`，不用关心函数实际放在哪一行。

### 6.2 `python/cpi_research/updater.py`：总览

这个文件是数据更新核心，可以拆成六台小机器：

1. 定义数据源和要取的 CPI 系列；
2. `_request_json` 负责联网；
3. `fetch_bls_series` 负责把 BLS 原始格式清洗成统一格式；
4. `_percent_change` 等小函数负责计算；
5. `_build_report` 和 `build_snapshot` 负责写报告；
6. `update_snapshot` 负责决定是否落盘。

#### 第 1～29 行：准备工具和系列清单

① 效果：告诉 Python 会用到哪些自带工具，以及需要向 BLS 索取哪些数据。

② 每行/每组作用：

- 第 1 行：文件说明；
- 第 3 行：允许使用较新的类型写法；
- 第 5～14 行：从 Python 工具箱取出 JSON、数学、文件、时间、网址请求和错误处理工具；
- 第 16 行：保存 BLS API 地址；
- 第 18～29 行：`SERIES` 登记表。左边如 `headline_sa` 是项目内部简称，`id` 是 BLS 官方系列编号，`name` 是给人看的中文名；
- `sa` 表示季调，`nsa` 表示未季调。

③ 运行后：程序知道要一次请求总 CPI、核心 CPI、住房、能源、食品、汽油、租金和二手车等十个序列。

#### 第 32～33 行：自定义错误 `BlsDataError`

① 效果：给“BLS 数据不可用”这类问题统一贴一个错误标签。

② 每行作用：

- `class` 可以理解为制作一种新标签模板；
- 它继承 `RuntimeError`，表示程序运行中的异常；
- 类里面只有说明文字，因为暂时不需要额外行为。

③ 运行后：上层脚本可以只捕获 `BlsDataError`，向用户显示友好的“CPI 更新失败”，而不是一大串内部错误。

#### 第 36～56 行：`_request_json`

① 效果：把请求寄给 BLS，最多尝试三次，并把返回内容读成 Python 登记表。

② 逐行作用：

- 第 36 行：函数接收请求内容、30 秒超时和 3 次尝试次数；开头下划线表示只打算在当前模块内部使用；
- 第 37～42 行：制作 HTTP 请求。地址是 BLS，正文把 Python 字典转成 JSON 字节，头部说明内容是 JSON，请求方法使用 POST；
- 第 43 行：准备一个盒子记录最后一次错误；
- 第 44 行：按尝试次数循环；
- 第 45～47 行：尝试打开网址，成功就直接读取 JSON 并返回；
- 第 48～51 行：遇到 HTTP 错误时，普通 4xx 通常是请求写错，立即失败；429 是请求太频繁，可以重试；
- 第 52～53 行：网络断开、超时或 JSON 损坏时记录错误；
- 第 54～55 行：如果还有机会，分别等待 1 秒、2 秒再重试；
- 第 56 行：三次都失败，抛出带原因的统一错误。

③ 运行后：临时网络抖动不会马上让定时任务失败；持续故障会返回清晰错误。

#### 第 59～102 行：`fetch_bls_series`

① 效果：向 BLS 请求原始序列，把不同序列整理成统一的 `{date, value}` 月度清单。

② 逐行作用：

- 第 59～64 行：函数入口，接收开始年、结束年、可选 API Key 和超时；
- 第 66～70 行：制作请求正文。列表推导式遍历 `SERIES`，收集所有官方 ID；
- 第 71～72 行：如果用户提供 Key，就放进请求；没提供也可以用公开额度；
- 第 74 行：真正联网；

## 13. 常见报错和解决办法

### 13.1 `node:fs/promises` 没有 `glob`

原因：正在使用 Node 20。

```bash
source ~/.zshrc
which node
node --version
```

应看到 `/opt/homebrew/opt/node@22/bin/node` 和 v22。如果不是：

```bash
export PATH="/opt/homebrew/opt/node@22/bin:$PATH"
```

### 13.2 `command not found: npm`

原因：Node 路径没生效。执行：

```bash
source ~/.zshrc
export PATH="/opt/homebrew/opt/node@22/bin:$PATH"
```

### 13.3 `Cannot find module` 或依赖损坏

先重新按锁文件安装：

```bash
npm ci
```

如果仍异常，可把旧依赖目录移走后重装：

```bash
mv node_modules node_modules.backup
npm ci
```

确认恢复后再自行删除备份。

### 13.4 端口 3000 已被占用

说明另一个开发服务还在运行。回到旧终端按 `Control + C`；或使用其他端口：

```bash
npm run dev -- --port 3001
```

### 13.5 `listen EPERM ... 9229`

如果只在受限沙箱里出现，是环境禁止监听调试端口。普通本机终端通常不会出现。请在 macOS Terminal 中运行 `npm run dev`。

### 13.6 BLS 请求超时、TLS 或 `429`

- 超时/TLS：通常是网络问题，更新器会自动重试三次；稍后再运行；
- 429：请求太频繁；等待后重试，或配置 BLS API Key；
- GitHub Actions 失败：打开失败步骤查看错误，确认 Actions 有 `contents: write` 权限。

### 13.7 `CPI series has fewer than 13 monthly observations`

某个系列无足够数据，不能计算同比。检查 BLS 系列 ID 是否有效、请求年份是否覆盖至少 13 个月。

### 13.8 `do not share a common release month`

某些分项尚未发布到同一个月。程序主动停止，是为了避免混用不同报告期。应等待数据齐全，不要绕过检查。

### 13.9 `Existing snapshot is invalid JSON`

手工修改 `latest.json` 时漏了逗号、引号或括号。查看 Git 差异并修复，或恢复正确版本后重新更新。

### 13.10 页面数据更新了但浏览器没变

依次尝试：

1. 浏览器强制刷新；
2. 停止并重新运行 `npm run dev`；
3. 确认 `lib/cpi/data/latest.json` 真有变化；
4. 线上环境确认新提交已触发重新部署。

### 13.11 GitHub 机器人无法 push

检查仓库 Actions 写权限和 main 分支保护。如果 main 禁止机器人直接推送，需要改成由工作流创建 Pull Request，而不是直接 push。

### 13.12 Homebrew 下载到一半失败

例如看到 `Failed to download resource "icu4c@78"`，通常是 GitHub 下载中断，不是 Node 不兼容。可以单独重试失败依赖：

```bash
HOMEBREW_NO_AUTO_UPDATE=1 brew fetch --retry icu4c@78
HOMEBREW_NO_AUTO_UPDATE=1 brew install node@22
```

`--retry` 最多重试五次并逐渐延长等待。安装成功后执行 `source ~/.zshrc`。

## 15. 项目核心代码提炼

整套系统最关键的五段逻辑是：

### 15.1 百分比变化

```text
(当前指数 ÷ 以前指数 - 1) × 100
```

它同时支撑环比和同比，只是“以前”分别选上月或去年同月。

### 15.2 降温规则

```python
cooling = core_mom < 0.25 and core_yoy < 3.0
```

这是项目观点形成的主要岔路口。

### 15.3 共同月份检查

```python
common_dates = set.intersection(...)
release_month = max(common_dates)
```

它保证不同分项不会混用报告月份。

### 15.4 数据没变就不写

```python
if _comparable(current) == _comparable(snapshot):
    return False
```

它保证自动化不会每天制造垃圾提交。

### 15.5 数据驱动页面

```tsx
{cpiReport.metrics.map((metric) => (
  <article key={metric.label}>...</article>
))}
```

页面不是手写五遍卡片，而是用数据清单自动生成。

## 17. 新手高频疑问

### Q1：Node、npm 和 React 是一回事吗

不是。Node 是发动机，npm 是依赖商店和命令工具，React 是网页积木库。

### Q2：Python 和 TypeScript 为什么同时存在

Python 擅长数据抓取和批处理；TypeScript 负责网页和类型检查。两边通过 `latest.json` 交接。

### Q3：修改 JSON 就等于修改数据库吗

不是。JSON 是随代码发布的历史种子，D1 是部署后的持久数据库。页面会把 JSON 历史同步到 D1，D1 不可用时才退回 JSON。

### Q4：`npm run dev` 会更新 BLS 数据吗

不会。它只启动网页。更新数据要运行 `npm run data:update`，或等待 GitHub 定时任务。

### Q5：GitHub 定时任务运行时，我的电脑需要开机吗

不需要。任务运行在 GitHub 的云端机器上。

### Q6：为什么页面代码中有很多括号和尖括号

尖括号描述网页结构；花括号插入 JavaScript 数据；圆括号把多行内容组成一个整体；大括号也用来界定函数和对象范围。

### Q7：`export` 和 `import` 是什么

`export` 是把工具放到公共货架；`import` 是另一个文件从货架取工具。

### Q8：为什么错误要主动 `throw`

宁可明确停止，也不要用错误或错月数据生成看似正常的交易报告。

### Q9：测试通过是否代表投资判断正确

不代表。测试证明代码符合既定公式和规则，不证明规则一定能预测市场。

### Q10：可以直接删除看不懂的配置吗

不要。构建配置通常在幕后生效。先用 Git 分支、测试和小步提交，再做删除。

### Q11：`class` 在这个项目里重要吗

目前只用自定义错误类和 Python 测试类。可以先把类理解成“同类物品的制作模板”，不必立即学习复杂面向对象知识。

### Q12：接口和 `interface` 是同一个意思吗

中文容易混淆：BLS API 接口是程序通信窗口；TypeScript `interface` 是数据形状说明书。两者不是同一个概念。

### Q13：当前数据一定是最新的吗

要同时查看 `latest.json` 的 `releaseMonth`、GitHub Actions 最近运行结果和线上部署时间。自动化存在不等于每次都成功。

### Q14：下一步最安全的小练习是什么

修改首屏的一句静态说明或一个 CSS 颜色，观察热更新；不要一开始改计算阈值、API ID或 Worker 配置。

## 18. 最终记忆图

```text
BLS 官方数据
  → Python 抓取与计算
```

1. 这份文件的输入来自哪里？
2. 它做了什么转换？
3. 输出交给谁？
4. 出错时谁负责拦住？

能回答这四个问题，就不是在死记代码，而是在真正理解系统。
